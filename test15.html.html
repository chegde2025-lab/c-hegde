2. OpenReserveSet

Detailed Description:
OpenReserveSet represents the insurer’s active outstanding financial liability. It is generated at exposure level because reserves are maintained by coverage type. This feed is consumed by actuarial systems, finance reporting platforms, and reinsurance partners. Only exposures that are financially active are included.

Business Rules:

  • Claim must be OPEN
  • Exposure must be OPEN
  • Reserve status must be OPEN
  • Outstanding amount must be greater than zero

SQL

SELECT c.claim_number, e.exposure_id,
SUM(r.reserve_amt) - SUM(r.paid_amt)
- COALESCE(SUM(rec.recovery_amt),0) AS outstanding
FROM claims c
JOIN exposure e ON c.claim_id = e.claim_id
JOIN reserve r ON e.exposure_id = r.exposure_id
LEFT JOIN recovery rec ON e.exposure_id = rec.exposure_id
WHERE c.status='OPEN'
AND e.status='OPEN'
AND r.status='OPEN'
GROUP BY c.claim_number, e.exposure_id
HAVING outstanding > 0;

XML

<OpenReserveSet>
   <Exposure Claim="CLM1001" ExposureID="10001" Outstanding="300000"/>
   <Exposure Claim="CLM1001" ExposureID="10002" Outstanding="200000"/>
   <Exposure Claim="CLM1002" ExposureID="10003" Outstanding="225000"/>
</OpenReserveSet>

XML Generation Logic: Each SQL row becomes one <Exposure> node under root element.

3. CloseReserveSet

Detailed Description:
CloseReserveSet communicates exposures that no longer carry financial liability. It ensures downstream systems stop reserve tracking and financial reporting for closed exposures.

SQL

SELECT c.claim_number, e.exposure_id
FROM claims c
JOIN exposure e ON c.claim_id = e.claim_id
JOIN reserve r ON e.exposure_id = r.exposure_id
WHERE c.status='CLOSED'
GROUP BY c.claim_number, e.exposure_id
HAVING SUM(r.reserve_amt) - SUM(r.paid_amt)=0;

XML

<CloseReserveSet>
   <Exposure Claim="CLM1003" ExposureID="10004" Outstanding="0"/>
</CloseReserveSet>

4. ClaimChangedSet

Detailed Description:
ClaimChangedSet is a delta feed that sends only claims modified after last execution. It prevents full data reload and improves system performance.

SQL

SELECT DISTINCT claim_number
FROM claims
WHERE last_update_ts > :last_run_time;

XML

<ClaimChangedSet>
   <Claim ClaimNumber="CLM1001"/>
</ClaimChangedSet>

5. Great American Insurance Set

Detailed Description:
This is a carrier-specific hierarchical feed that includes claim-level, exposure-level, and reserve-type-level breakdown. It provides a complete financial view for integration with external carrier systems.

SQL

SELECT c.claim_number, e.exposure_id, r.reserve_type,
SUM(r.reserve_amt) - SUM(r.paid_amt)
- COALESCE(SUM(rec.recovery_amt),0) AS outstanding
FROM claims c
JOIN exposure e ON c.claim_id = e.claim_id
JOIN reserve r ON e.exposure_id = r.exposure_id
LEFT JOIN recovery rec ON e.exposure_id = rec.exposure_id
GROUP BY c.claim_number, e.exposure_id, r.reserve_type;

XML

<GreatAmericanInsuranceSet>
   <Claim ClaimNumber="CLM1001">
      <Exposure ID="10001">
         <Reserve Type="Indemnity" Outstanding="250000"/>
         <Reserve Type="Expense" Outstanding="50000"/>
      </Exposure>
   </Claim>
</GreatAmericanInsuranceSet>

XML Logic: Nested grouping by claim → exposure → reserve type.

6. CompanionClaimChangeSet

Detailed Description:
CompanionClaimChangeSet identifies related claims that share common attributes. Claims are considered companions if they share:

  • Same Incident ID
  • OR Same Loss Date
  • OR Same Loss Place (PlaceRole → Place)

This ensures all related claims are evaluated together for litigation, fraud detection, and financial exposure aggregation.

SQL Using OR Logic

SELECT DISTINCT c.claim_number
FROM claims c
LEFT JOIN vehicleincident vi ON c.claim_id = vi.claim_id
LEFT JOIN placerole pr ON c.claim_id = pr.claim_id
WHERE vi.incident_id IN (
    SELECT incident_id FROM vehicleincident GROUP BY incident_id HAVING COUNT(*) > 1
)
OR c.loss_date IN (
    SELECT loss_date FROM claims GROUP BY loss_date HAVING COUNT(*) > 1
)
OR pr.place_id IN (
    SELECT place_id FROM placerole GROUP BY place_id HAVING COUNT(*) > 1
);

XML

<CompanionClaimChangeSet>
   <Claim>CLM1001</Claim>
   <Claim>CLM1002</Claim>
</CompanionClaimChangeSet>

XML Generation Logic: SQL identifies all companion claims using OR conditions. Each qualifying claim becomes one <Claim> node under root element.
